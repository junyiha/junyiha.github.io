MFC BEGIN_MESSAGE_MAP 详解

在 MFC 中，消息映射（Message Map）机制是把 Windows 消息（如鼠标点击、按键、绘制等）“路由”到相应的成员函数的核心手段。而 BEGIN_MESSAGE_MAP／END_MESSAGE_MAP 宏正是构建这套机制的起点和终点。下面从宏的定义、运行时展开过程，到常见用法与注意事项，做一个详细拆解。

一、宏的基本语法

  • theClass：当前要实现消息映射的 MFC 类（通常是 CWnd 派生类，如对话框、视图、控件等）。
  • baseClass：父类的消息映射类名。MFC 会先在当前类查找映射条目，找不到时再沿着 baseClass 链向上查找。

二、宏展开原理（精简版）

在头文件 afxmsg_.h（或者 afxwin.h）中，大致定义如下（已简化，省略少量细节）：

  • **_messageEntries[]**：当前类所有消息—函数映射的静态数组，最后一个元素是“终止标记”。
  • messageMap：包含两指针——指向父类的 messageMap（实现继承），以及指向本类的 _messageEntries 数组起始位置。
  • **GetMessageMap()**：虚函数，用于在运行时获取当前对象的映射表。

MFC 在运行时，调用 CWnd::WindowProc（或特定消息处理入口）时，会沿着 GetMessageMap() 链遍历这些表，匹配合适的消息签名，然后调用对应成员函数。

三、常见映射条目举例

宏名 功能 示例
ON_BN_CLICKED(id, func) 按钮点击 (BN_CLICKED) ON_BN_CLICKED(IDC_OK, &CMyDlg::OnOKClicked)
ON_WM_PAINT() WM_PAINT 消息 ON_WM_PAINT()
ON_WM_LBUTTONDOWN() 鼠标左键按下 (WM_LBUTTONDOWN) ON_WM_LBUTTONDOWN()
ON_COMMAND(id, func) 菜单、工具栏命令、控件通知 ON_COMMAND(ID_FILE_OPEN, &CMyView::OnFileOpen)
ON_NOTIFY(nmhdrID, id, func) 通知消息（如 NM_CLICK 等） ON_NOTIFY(NM_CLICK, IDC_LIST, &CMyDlg::OnListClick)

这些宏最终都会生成一个 AFX_MSGMAP_ENTRY 条目，包含消息类型、控制 ID、函数指针等信息。

四、使用注意与最佳实践

  1. 配套声明

    • 在类头文件中必须写 DECLARE_MESSAGE_MAP()。
    • 在实现文件中才能用 BEGIN_MESSAGE_MAP／END_MESSAGE_MAP。

  2. 继承链

    • BEGIN_MESSAGE_MAP(MyView, CView) 会保证：若 MyView 未处理某消息，会自动转交给 CView（再往上到 CWnd）处理。

  3. 签名匹配

    • 不同消息对应的成员函数签名必须严格一致（如 afx_msg void OnPaint()、afx_msg void OnLButtonDown(UINT nFlags, CPoint point) 等）。

  4. ON_CONTROL_REFLECT

    • 如果在控件派生类内部自行处理通知消息，用 ON_CONTROL_REFLECT 而不是 ON_XXX，这样消息会“反射”回来，让控件自己先处理。

  5. 顺序和覆盖

    • 同一消息若在子类和父类都有映射，子类的映射优先。若想屏蔽父类实现，可在子类条目中返回即可。

  6. 自定义消息

    • 对于自定义的 Windows 消息（#define WM_MYMESSAGE (WM_USER+100)），需要使用 ON_MESSAGE(WM_MYMESSAGE, &MyClass::OnMyMessage)，对应函数签名为 afx_msg LRESULT OnMyMessage(WPARAM, LPARAM)。

五、完整示例

关键点回顾：

  1. DECLARE_MESSAGE_MAP() + BEGIN_MESSAGE_MAP/END_MESSAGE_MAP 共同生成并链接消息表。
  2. 映射条目宏（ON_BN_CLICKED、ON_WM_PAINT 等）插入数组 _messageEntries。
  3. 运行时通过 GetMessageMap() 获取表，并逐级查找、调用对应处理函数。

通过上述拆解，你应该对 BEGIN_MESSAGE_MAP 的作用、宏展开的内部结构，以及如何正确书写消息映射有了全面的理解。若还有具体的宏定义细节或 “为什么这么写” 的疑问，欢迎继续讨论！

MFC cmake编译都需要那些编译参数

在用 CMake 构建 MFC 应用时，除了常规的 C++ 编译器和链接选项以外，最关键的几个“MFC 特定”参数如下：

  1. CMAKE_MFC_FLAG
    控制 MFC 的启用及链接方式（目录级变量）：

MFC cmake编译 详解

下面给出一个从零开始，使用 CMake 构建 MFC（Microsoft Foundation Classes）应用的详解，包括常见设置、变量说明、示例 CMakeLists.txt 以及动态／静态链接两种模式的区别。

一、前提条件

  1. CMake 版本
    推荐使用 CMake 3.8 或更高版本，因为较新版本在生成 VS 工程时对 MFC 的支持更完善。

  2. Visual Studio 生成器

    • 只能在 Windows 平台下使用 -G "Visual Studio XX YYYY"（如 -G "Visual Studio 17 2022"）或新版 CMake 的 -G "Visual Studio 17 2022"。
    • CMake 会根据指定的架构（-A x64 / -A Win32）生成对应的项目。

  3. 项目类型
    典型的 MFC 应用有两种：

    • 基于对话框（Dialog‑Based）
    • 基于文档／视图（Doc‑View）
      本例以最简单的对话框程序为例。

二、核心变量与属性

名称 作用
CMAKE_MFC_FLAG 决定是否以及如何使用 MFC：
0=不启用，1=静态链接，2=动态链接（DLL）
WIN32 在 add_executable 中传入，告诉 CMake 生成 Windows GUI 应用，而非 Console
add_definitions(-D_AFXDLL) 当 CMAKE_MFC_FLAG=2（动态 MFC）时，需要定义此宏，启用 MFC DLL 运行时
target_compile_definitions 推荐方式，为单个 target 添加宏定义
target_link_libraries 链接 MFC 依赖库（通常不需要手动写，CMake 会根据 CMAKE_MFC_FLAG 自动处理）

MFC cmake CMAKE_MFC_FLAG 是什么

在 CMake 中，CMAKE_MFC_FLAG 是一个专门用来控制生成的 Visual Studio 工程是否（以及如何）使用 Microsoft Foundation Classes (MFC) 的变量：

  • 取值含义

    • 0（默认）——不使用 MFC。
    • 1——使用静态链接的 MFC 库（静态 CRT），即把 MFC 代码编译进你的可执行文件或 DLL。
    • 2——使用**共享（动态）**的 MFC 库（DLL 形式），此时通常还需要加上 -D_AFXDLL 来启用运行时 DLL 版 MFC。(cmake.org, stackoverflow.com)

  • 作用范围
    CMAKE_MFC_FLAG 是目录级别变量，凡是在同一目录中定义的所有目标（add_executable / add_library）都会受到它的影响，无法直接对单个目标分别设置。如果需要在不同目标间切换，建议将它们分放在不同的子目录中，各自设置不同的值。(discourse.cmake.org)

  • 典型用法

MFC TEXT宏 详解

在 Windows API（包括 MFC）中，TEXT（在 <tchar.h> 中定义）是一个宏，用来让字符串字面量根据编译设置自动适配为 ANSI（多字节）或 Unicode（宽字符）形式。它常与 _T 宏等价，用于实现“可移植”字符编码。以下从原理、定义、用法及注意事项来详细说明。

1. 为什么需要 TEXT 宏

  • 历史背景
    早期 Windows 仅支持 ANSI（即多字节编码）API，字符串常写作 "Hello"。
    随 Windows 98/NT 之后，微软引入了 Unicode（宽字符）API，对应的函数和类型以 W 结尾（如 CreateWindowW、LPWSTR）。

  • 兼容性需求
    为了让同一份代码能在 ANSI 和 Unicode 两种模式下编译，Windows SDK 提供了两个版本的 API（ANSI 版本以 A 结尾，Unicode 版本以 W 结尾），以及一套“泛型”宏和类型（无后缀），例如：

    同理，字符串文字也需要对应切换。

2. 宏定义

  • 在 <tchar.h> 中：

    • 当定义了 _UNICODE（或 UNICODE）时，TEXT("Foo") → L"Foo"（宽字符字面量，类型 const wchar_t*）。
    • 否则，TEXT("Foo") → "Foo"（窄字符字面量，类型 const char*）。

5. 注意事项

  • 源码编码
    在 Unicode 模式下，源文件应确保保存为支持 Unicode 字符的编码（如 UTF‑8 带 BOM 或 UTF‑16），否则在字面量中写入非 ASCII 字符时可能乱码。

  • 宏与原始字面量混用
    不要将未经过 TEXT 或 L 前缀的窄字面量传给 Unicode API，也不要将宽字符字面量（`L”…”）传给 ANSI API，否则会产生类型不匹配或运行时错误。

  • 现代替代
    在 C++11 及更高版本，建议直接使用标准宽字符字面量 u8"…", u"…", U"…", 或 char16_t/char32_t，以及 std::wstring，并直接调用对应的 Unicode API；如果只针对现代 Windows 平台，也可直接定义项目为 Unicode，无需依赖 TEXT 宏。

6. 小结

  • **TEXT**（或 _T）宏根据编译时是否定义 _UNICODE 自动在字符串字面量前添加 L 前缀，保证代码在 ANSI/Unicode 模式下通用。
  • 通常与 TCHAR、LPCTSTR 等“泛型”类型配合使用，以编写与字符集无关的 MFC/Win32 代码。
  • 在现代 C++/Windows 开发中，如果项目完全以 Unicode 为目标，也可直接使用宽字符类型和字面量，减少宏层的复杂度。
